# CLAUDE.md

このファイルは Claude Code がこのリポジトリで作業する際の前提知識・方針をまとめたもの。実装前に必ず目を通すこと。

## プロジェクトの目的

Nintendo Switch Sports「サッカー」のプレイ映像をキャプチャーボード経由でリアルタイム解析し、勝敗・ランク変動を自動検知してSQLiteに記録する。将来的には配信画面にグラフを表示する。

現在のスコープは「勝敗」「ランク(∞帯の数値・ゲージ)」に加え、「ゴール・アシスト(得点者・アシスト者名)」の記録。ゴール・アシストの記録は、プレイヤー許可リスト(`.env`の`ALLOWED_PLAYERS`)にある得点者のもののみ記録し、それ以外は記録しない(他プレイヤーのプライバシー配慮)。

## 確定している設計方針

### 実行環境

- Windows のみを想定(Mac/Linux対応は考えなくてよい)
- キャプチャーボード: I-O DATA GV-USB3HDS/E、キャプチャ解像度は1920x1080を想定

### 映像取得の構成

- Python側は物理キャプチャーデバイスを直接開かない。OBS Studio の **Virtual Camera** 出力を映像ソースとして利用する
  - 理由: UVCデバイスは基本的に単一アプリしか掴めないため、OBSが配信・録画のために専有している間もPythonが同じ映像を取得できるようにするため
- リアルタイム処理を行う(録画ファイルを後から処理する方式ではない)
- **取得方法: `cv2.VideoCapture`は使わない。`ffmpeg`をサブプロセスとして起動し、`dshow`経由で`"OBS Virtual Camera"`から生フレーム(bgr24, rawvideo)を標準出力経由で読み取る**
  - 動作確認済み。`cv2.VideoCapture`(MSMF・DSHOWいずれのバックエンドでも)はOBS Virtual Cameraからのフレーム取得に失敗することを実機で確認済み。これはOpenCVとOBS標準Virtual Cameraの間で長年報告されている既知の相性問題であり、この環境固有の設定ミスではない
  - デバイス名は`"OBS Virtual Camera"`(`ffmpeg -list_devices true -f dshow -i dummy`で確認済み)
  - 依存パッケージ: `imageio-ffmpeg`(ffmpeg本体を同梱)。`opencv-python`は画像処理(検知ロジック)側でのみ使用し、キャプチャには使わない
  - 解像度は1920x1080固定で読み取る(OBSの出力解像度と一致させること)
  - 継続的な読み取り・フレーム抜け対策は`capture/ffmpeg_capture.py`の`FfmpegFrameReader`で実装済み。バックグラウンドスレッドで読み取り続け、処理が追いつかない間の古いフレームは破棄して常に最新フレームを返す設計。ただし**OBS Virtual Camera実機(dshow)での疎通確認はまだ行っていない**(テストは`fixtures/videos/`の実動画ファイルを入力に差し替えて検証。`scripts/capture_stream_test.py`で実機確認すること)

### 検知方式の方針

- 勝敗判定: 画面上部に出る斜めの結果バナーの**色・形状**で判定する(OCRではない)。実測では勝ち=緑〜ティール系、負け=黒系の配色で、チームカラー(青/赤/ピンク)には依存しない
- ランク数値: 画面左下の固定領域をOCRで読み取る(EasyOCR等を想定)。実装済み(`detection/rank_ocr.py`)
  - 帯番号(整数)はOCR、次の帯までのゲージの溜まり具合(0.0〜1.0)はランクバッジ下部の横長ゲージのHSV明度(Value)を色ベースで判定し、`帯番号 + 溜まり具合`の小数値として組み合わせる(`read_precise_rank`)。DBにはこの小数値をそのまま記録し、丸め(四捨五入等)は将来の表示層の責務とする
  - ランクバッジは表示タイミングによって明確に異なる2つのサイズで描画される(結果バナー確定直後の**コンパクト表示**と、ランク変動アニメーション開始後〜暗転までの**拡大表示**)。ゲージの塗りつぶし「割合」は列単位のピクセル計測のため、1つの固定ROIで両方を賄おうとすると実際のバー端とずれて誤った割合を返す(実データで判明、勝敗で塗りつぶし量が変わるはずの前後ペアが同じ値を返す不具合があった)。そのため`GAUGE_ROI_COMPACT`/`GAUGE_ROI_ENLARGED`を個別に用意し、`read_precise_rank(frame, gauge_roi)`は呼び出し元が明示的にどちらを使うか指定する(`state/match_state.py`では、結果バナー確定直後の読み取り=常にコンパクト、ランク変動安定後の読み取り=常に拡大、と呼び出しタイミングで一意に決まる)
  - リーグ昇格/降格の判定には必ず帯番号(整数)側を使うこと。小数のランク値同士で比較すると、帯自体は変わっていないのにゲージの僅かな増減だけで誤検知するため(`state/match_state.py`参照)
- 状態監視(アニメーション中かどうかの判定)は固定領域のピクセル差分・平均色で行う。軽量な処理なので高頻度に回してよい

### サンプリング戦略(重要)

- 「軽量な監視」(色判定・ピクセル差分)はキャプチャの生フレームレートのまま(30fps想定)常時実行してよい。負荷はほぼ無視できる
- 「重い処理」(OCR)は常時実行しない。軽量監視により「数値が変化しなくなった=確定」を検知した瞬間に**1回だけ**発火させる
- この2層構成(常時軽量監視 + イベント駆動の重い処理)を崩さないこと

### 試合後の状態遷移

1. 結果バナー表示 → 勝敗+開始時ランクを記録
2. ランク変動アニメーション → 数値が安定するまで監視(OCRは確定時のみ)
3. ランク確定 → 最終値、リーグ昇格/降格の有無を記録
   - リーグ昇格/降格の演出画面は、ランクの帯(数値)が実際に変化した場合のみ表示される特別イベント。通常はゲージの増減のみ
4. 暗転 → ここでクールダウン(この試合の記録完了とみなす)
5. マッチング画面 → 次の試合の検知を再開

### `src/` のモジュール構成

```
src/
└── nss_tracker/
    ├── capture/       # ffmpegサブプロセス起動・生フレームの継続読み取り/バッファリング
    ├── detection/      # 画像解析ロジック(banner.py: 勝敗バナー判定, rank_ocr.py: ランクOCR, motion.py: ピクセル差分監視)
    ├── state/          # 試合の状態遷移(バナー表示→ランクアニメ→確定→暗転→マッチング)の管理
    ├── database/       # SQLiteへの読み書き
    └── web/            # 将来のグラフ表示用(今は空、境界だけ確保)
```

- `detection/`は当面`banner.py` / `rank_ocr.py` / `motion.py` / `league_change.py` / `goal.py`のフラット構成とする。追加OCRなど将来の機能追加が必要になった段階で、その都度サブディレクトリに整理し直してよい(先回りして細分化しない)

### ゴール・アシストの記録とプレイヤー許可リスト

- 得点者・アシスト名は`detection/goal.py`でOCR読み取りする。数字専用の`rank_ocr.py`とは異なり任意の日本語混じり文字列を読む必要があるため、EasyOCR(信頼度0.00〜0.16程度でほぼ使い物にならなかった)ではなく**PaddleOCR**(`lang="japan"`)を使う。実測では16件中15件(94%)を正しく読み取れた
- プレイヤー許可リスト: `.env`(`.gitignore`対象、他プレイヤーの実名を含むため)の`ALLOWED_PLAYERS`にカンマ区切りで書かれたプレイヤーのみ、得点・アシストを記録する。リストに無いプレイヤーが得点者の場合はそのゴールを記録すらしない(`database/db.py`の`save_goal`が担当)。テンプレートは`.env.example`(git追跡対象)を参照すること
- アシスト者名は、得点者が許可リストにいれば、アシスト者自身の許可リスト有無に関わらずそのまま記録する
- 許可リストによるフィルタリングは永続化層(`database/db.py`)の責務とし、検知層(`detection/`)・状態機械(`state/match_state.py`)はポリシーを持たず見えたものをそのまま報告する
- ゴール検知はWATCHING状態(結果バナーを待っている=プレイ中)の間のみ有効。試合単位でメモリ上にバッファし、試合結果確定時に`MatchResult.goals`として払い出す

### テスト方針

- `tests/`に`pytest`を導入し、`fixtures/screenshots/*.png`を入力に`detection/`配下の関数(banner判定・OCR・motion監視)を検証する
- ファイル名は`docs/screen_states.md`の命名規則に対応させ、パラメータ化テストで期待値(勝敗・ランク値など)を検証する
- `fixtures/screenshots/`は`.gitignore`対象でリモートに含まれないため、**fixtureディレクトリが存在しない環境ではこれらのテストを自動でskipする**(`pytest.mark.skipif`等でディレクトリの有無をチェック)
- 状態遷移(`state/`)やDB読み書き(`database/`)など実画像に依存しないロジックは合成データ/モックでテストし、CI・クローン直後でも常に実行できるようにする
- OCR(EasyOCR)は重い処理のため`@pytest.mark.slow`などで分離し、デフォルトの`pytest`実行では回さず明示的に指定したときのみ実行する

### ログ方針

- 標準の`logging`モジュールを使う(`print()`は使わない)。設定は`app.py`(エントリーポイント)で一箇所にまとめる
- 出力先はコンソール(標準出力)+ローテーティングファイル(`logs/tracker.log`、`.gitignore`の`*.log`で除外対象)の両方
- ログレベルの使い分け:
  - `DEBUG`: フレーム単位・ピクセル差分などの高頻度な内部状態(デフォルトでは出さない)
  - `INFO`: 試合ライフサイクルの節目(バナー検知、ランク確定、記録完了など)
  - `WARNING`: 想定外の状態(フレーム取得タイムアウト、想定外の画面遷移など)
  - `ERROR`: 復旧不能な異常(ffmpegプロセスの異常終了など)
- レベル切り替えは環境変数か定数1つで簡単に済ませる(凝った設定ファイルは今は不要)

### 常駐アプリとしての起動方法

- 現段階では手動でターミナルから`uv run python main.py`を実行する運用のみを想定し、タスクスケジューラ等の自動起動・自動復旧の仕組みは作らない
- 実際に配信で運用してみて自動復旧などの必要性が具体化した段階で改めて検討する
- `main.py`でcapture(`FfmpegFrameReader`)→state(`MatchStateMachine`)→database(`db`)の配線を実装済み。OBS/Switchが無い段階でも配線を確認できるよう、`--video path/to/file.mp4`で動画ファイルをOBS Virtual Cameraの代わりに読み込める(動作確認は完了、実機での疎通確認は別途`docs/capture_verification.md`参照)
  - `MatchStateMachine`のフレーム数ベースの閾値(`banner_confirm_frames`等)は30fps想定のデフォルト値のため、呼び出し側(`main.py`)で実際のfpsに応じてスケーリングする必要がある(`state/match_state.py`のdocstring参照)。`--video`指定時はファイルのfpsを自動検出、実キャプチャ時は30fps想定
  - EasyOCR・PaddleOCRは初回呼び出し時にモデル読み込みで数秒かかる(実測: 約3.8秒、2回目以降は0.2秒未満)。キャプチャループ開始前に一度呼び出して済ませておかないと、ちょうどランクバッジの安定待ちが始まる直後にこの数秒間が重なり、`FfmpegFrameReader`の「追いつかない間は古いフレームを破棄する」設計と組み合わさって肝心の区間のフレームを丸ごと読み飛ばしてしまう(`main.py`の`_warmup_ocr_engines`で対応済み)

## フォルダの位置づけ

- `scripts/`: 人が手動で実行する診断・検証用スクリプト置き場(自動テストではない)。例: `list_dshow_devices.py`(dshowデバイス名の確認)、`capture_test_frame.py`(OBS Virtual Cameraからの1フレーム取得確認)。実機検証で判明した知見をここに残しておくことで、同じ調査を繰り返さないようにする

## 開発フロー(Git運用)

詳細は [`docs/git_workflow.md`](docs/git_workflow.md) を参照。要点は以下の通り、必ず守ること。

- `main`へ直接pushしない。タスクごとに`feature/xxx`・`fix/xxx`ブランチを切って作業する
- 1タスク=1ブランチ=1PRを基本とし、`main`へのマージ前に必ずPRを作成する
  - 1タスクが大きくなるときは、親ブランチ`feature/xxx`を切り、子ブランチ`feature/xxx/yyy`をそこにマージし、最後に`main`ブランチへマージする
- コミットメッセージはConventional Commits形式(`feat:`, `fix:`, `docs:` など)でこまめに行う
- PRは見出し3以下を使用し、動作確認内容とドキュメント更新の有無を必ずチェックボックスで含める
- 新たな実装をしたり、決め事をした場合は、必要に応じて都度マークダウンファイルに追記する
- `main`・親ブランチへのforce-pushや履歴書き換え(`rebase -i`など)は行わない。必要な場合は必ずユーザーに確認する
- 上記「未確定・要相談」に該当する仕様に関わる実装は、着手前に必ずユーザーに確認を取ってから進める
