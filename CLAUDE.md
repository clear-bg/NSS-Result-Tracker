# CLAUDE.md

このファイルは Claude Code がこのリポジトリで作業する際の前提知識・方針をまとめたもの。実装前に必ず目を通すこと。

## プロジェクトの目的

Nintendo Switch Sports「サッカー」のプレイ映像をキャプチャーボード経由でリアルタイム解析し、勝敗・ランク変動を自動検知してSQLiteに記録する。将来的には配信画面にグラフを表示する。

現在のスコープは「勝敗」「ランク(∞帯の数値・ゲージ)」に加え、「ゴール・アシスト(得点者・アシスト者名)」の記録。ゴール・アシストの記録は、プレイヤー許可リスト(`.env`の`ALLOWED_PLAYERS`)にある得点者のもののみ記録し、それ以外は記録しない(他プレイヤーのプライバシー配慮)。

## 確定している設計方針

### 実行環境

- Windows のみを想定(Mac/Linux対応は考えなくてよい)
- キャプチャーボード: I-O DATA GV-USB3HDS/E、キャプチャ解像度は1920x1080を想定

### 映像取得の構成

- Python側は物理キャプチャーデバイスを直接開かない。OBS Studio の **Virtual Camera** 出力を映像ソースとして利用する
  - 理由: UVCデバイスは基本的に単一アプリしか掴めないため、OBSが配信・録画のために専有している間もPythonが同じ映像を取得できるようにするため
- リアルタイム処理を行う(録画ファイルを後から処理する方式ではない)
- **取得方法: `cv2.VideoCapture`は使わない。`ffmpeg`をサブプロセスとして起動し、`dshow`経由で`"OBS Virtual Camera"`から生フレーム(bgr24, rawvideo)を標準出力経由で読み取る**
  - 動作確認済み。`cv2.VideoCapture`(MSMF・DSHOWいずれのバックエンドでも)はOBS Virtual Cameraからのフレーム取得に失敗することを実機で確認済み。これはOpenCVとOBS標準Virtual Cameraの間で長年報告されている既知の相性問題であり、この環境固有の設定ミスではない
  - デバイス名は`"OBS Virtual Camera"`(`ffmpeg -list_devices true -f dshow -i dummy`で確認済み)。実キャプチャ時(`--video`未指定時)は`.env`の`CAPTURE_DEVICE_NAME`から読み込む(`config.py`経由)。フォールバック用のデフォルト値は持たず、未設定の場合は起動時に`ConfigError`で明示的に失敗する
  - 依存パッケージ: `imageio-ffmpeg`(ffmpeg本体を同梱)。`opencv-python`は画像処理(検知ロジック)側でのみ使用し、キャプチャには使わない
  - 解像度は1920x1080を想定(OBSの出力解像度と一致させること)。実キャプチャ時は同様に`.env`の`CAPTURE_WIDTH`/`CAPTURE_HEIGHT`が必須(未設定時は`ConfigError`)
  - 継続的な読み取り・フレーム抜け対策は`capture/ffmpeg_capture.py`の`FfmpegFrameReader`で実装済み。バックグラウンドスレッドで読み取り続け、処理が追いつかない間の古いフレームは破棄して常に最新フレームを返す設計。**OBS Virtual Camera実機(dshow)での疎通確認は2026-07-17に完了済み**(`scripts/capture_stream_test.py`、573フレーム/10秒・実測57.2fps、画質・色・アスペクト比とも問題なし。詳細は`docs/capture_verification.md`参照)

### 検知方式の方針

- 勝敗判定: 画面上部に出る斜めの結果バナーの**色・形状**で判定する(OCRではない)。実測では勝ち=緑〜ティール系、負け=黒系の配色で、チームカラー(青/赤/ピンク)には依存しない
  - 引き分け(draw)のバナー帯は負けと似た暗い配色のため、帯の色だけでは負けと区別できない。文字「引き分け」の縁取りに勝ちと同じミントグリーンが使われている点を追加の判定材料として区別する(`detection/banner.py`参照)。なお現時点の参照素材はランクを賭けない対戦の引き分けのみで、この場合ランクバッジ自体が表示されない(暗転演出も無く、バナー消灯後は直接メニュー画面に遷移する。バッジ非表示のwin/loseケースと同じ経路で処理される)。ランクを賭けた対戦での引き分け(バッジ表示あり)は参照素材が無く未検証のため、「引き分けは常にバッジが表示されない」と一般化しないこと
  - Issue #67対応: 実プレイ配信のアーカイブ映像で、通常プレイ中(ゴール演出とは無関係)にカメラアングルの都合でスタジアムの背景がBANNER_ROIに写り込み、1.3秒程度連続して"lose"と誤判定され、結果バナーの誤検知(1試合が2つに分割される)が発生した。この背景はIssue #45の建造物ケースと異なりHue標準偏差も本物のバナーと近く、既存のBANNER_HUE_STD_MAXフィルタでは区別できない。ゴール演出中だけ判定を止めるステートベースの案・HUD要素の有無や複数ROIでの整合性チェックといった追加の判定条件もいずれも実データ検証で本物のバナー確定を壊すことが判明したため不採用とした(本物のバナーは表示直後にアニメーションで変形し、確定に必要な連続区間の途中で条件を満たさなくなるため)。この種の誤検知の参照サンプルが現時点で1件のみで根本的な検知改善の閾値を決められるだけのデータが無いため、対症療法として`state/match_state.py`の`banner_confirm_frames`のみ1秒から2秒に延長した(`DEFAULT_BANNER_CONFIRM_FRAMES`・`main.py`の`_make_match_state_machine`参照。誤検知は1.3秒程度しか持続せず本物のバナーは数秒以上表示されるため、2秒あれば防げる)。回帰テストは`fixtures/videos/21_goal_event_false_positive_win_blue_4-3.mp4`(`fixtures/videos/metadata.json`)
  - Issue #76対応: Issue #67の2秒デバウンスによる検知遅延を緩和するため、試合の本当の終了時点にのみ表示される「試合終了」バナー(`detection/match_end.py`、`fixtures/screenshots/65_match_end.png`)を補助信号として使う。色ベースの軽量な候補判定(`is_match_end_screen`)が一定時間連続したタイミングで、OCR(PaddleOCR)による文字確認(`confirm_match_end_text`)を1回だけ行う。色味が酷似する「延長戦」「キックオフ」バナーとの誤認識を防ぐため2段構成にしている(色だけでは「延長戦」は帯の横幅の違いで除外できるが、「キックオフ」は除外できないため、確定的な判定には文字そのものを読む必要があると実データで判明した)。「試合終了」を確認できていればbanner_confirm_framesを短縮(Issue #67修正前と同じ1秒)、確認できていなければ従来どおり2秒のまま。「試合終了」検知自体を見逃す実データケース(fixtures/videos/21_goal_event_false_positive_win_blue_4-3.mp4のような、確認できた区間を含まない動画)があるため、「VS画面〜試合終了検知までバナー判定自体を止める」完全ゲート方式ではなく、見逃しても2秒デバウンスにフォールバックするだけで正しさは損なわれない補助信号方式を採用した(`state/match_state.py`のモジュールdocstring参照)
- ランク数値: 画面左下の固定領域をOCRで読み取る(EasyOCR等を想定)。実装済み(`detection/rank_ocr.py`)
  - 帯番号(整数)はOCR、次の帯までのゲージの溜まり具合(0.0〜1.0)はランクバッジ下部の横長ゲージのHSV明度(Value)を色ベースで判定し、`帯番号 + 溜まり具合`の小数値として組み合わせる(`read_precise_rank`)。DBにはこの小数値をそのまま記録し、丸め(四捨五入等)は将来の表示層の責務とする
  - ランクバッジは表示タイミングによって明確に異なる2つのサイズで描画される(結果バナー確定直後の**コンパクト表示**と、ランク変動アニメーション開始後〜暗転までの**拡大表示**)。ゲージの塗りつぶし「割合」は列単位のピクセル計測のため、1つの固定ROIで両方を賄おうとすると実際のバー端とずれて誤った割合を返す(実データで判明、勝敗で塗りつぶし量が変わるはずの前後ペアが同じ値を返す不具合があった)。そのため`GAUGE_ROI_COMPACT`/`GAUGE_ROI_ENLARGED`を個別に用意し、`read_precise_rank(frame, gauge_roi)`は呼び出し元が明示的にどちらを使うか指定する(`state/match_state.py`では、結果バナー確定直後の読み取り=常にコンパクト、ランク変動安定後の読み取り=常に拡大、と呼び出しタイミングで一意に決まる)
  - リーグ昇格/降格の判定には必ず帯番号(整数)側を使うこと。小数のランク値同士で比較すると、帯自体は変わっていないのにゲージの僅かな増減だけで誤検知するため(`state/match_state.py`参照)
  - Issue #73対応で`read_rank_tier(frame, roi=RANK_ROI)`を追加し、バッジのアイコン部分から`'∞'`/`'S'`/`'A'`/`None`を識別できる(`read_rank()`と同じROIをallowlistなしでOCRし、最も上側のテキストをアイコンとみなす方式。新規のROI測定は不要)。ただし∞帯は既存fixtureで実データ検証済みな一方、**S/A帯は結果バナー画面での参照fixtureが無く未検証**(vs_rank.py用のVS画面アイコンROIを代用して試したが、バッジが小さすぎてEasyOCRが文字自体を安定して検出できず代用にならないと判明した)。B/C/D/E帯・`match_state.py`/DBへの統合(戻り値の設計、league_changed判定への組み込み等)は未着手(Issue #43でS/A帯の結果バナーfixtureが集まり次第、実データで検証・着手する)
- VS画面(マッチング完了直後)の両チーム最大4人分のランクバッジは`detection/vs_rank.py`が別途読み取る(結果バナー側の`rank_ocr.py`とはROI・OCRエンジンとも別実装、Issue #39/#52)。∞帯に加え、Issue #40対応でS/A帯も識別できる(`SlotRank(tier, value)`で返す。`tier`は`'∞'`/`'S'`/`'A'`/`None`)。B/C/D/Eは参照fixtureがまだ無く未着手(Issue #43でfixture収集後に着手)
- 状態監視(アニメーション中かどうかの判定)は固定領域のピクセル差分・平均色で行う。軽量な処理なので高頻度に回してよい

### サンプリング戦略(重要)

- 「軽量な監視」(色判定・ピクセル差分)はキャプチャの生フレームレートのまま(30fps想定)常時実行してよい。負荷はほぼ無視できる
- 「重い処理」(OCR)は常時実行しない。軽量監視により「数値が変化しなくなった=確定」を検知した瞬間に**1回だけ**発火させる
- この2層構成(常時軽量監視 + イベント駆動の重い処理)を崩さないこと

### 試合後の状態遷移

1. 結果バナー表示 → 勝敗+開始時ランクを記録
2. ランク変動アニメーション → 数値が安定するまで監視(OCRは確定時のみ)
3. ランク確定 → 最終値、リーグ昇格/降格の有無を記録
   - 帯(数値)が実際に変化した場合のみ、通常のゲージ増減に加えて追加の演出が入ることがある。ただし**昇格と降格で演出の見た目が異なる**(実データで確認済み、`fixtures/videos/10_RankDown_red.mp4`参照):
     - 昇格: ランクバッジ自体が隠れる**全画面の半透明オーバーレイ**(`detection/league_change.py`の`is_league_change_screen()`が検知)。演出中はバッジが読み取れないため、`state/match_state.py`はこの演出が終わるまで確定を待つ
     - 降格: 全画面オーバーレイは出ない。ランクバッジの上に小さな「降格」ラベルが乗るだけで、バッジ自体はその間も読み取り可能。`is_league_change_screen()`はこのケースでは常にFalseを返すが、`state/match_state.py`はバナー消灯時点で「演出は来ない」と判断して確定するフォールバック経路(および緩やかな値変化を追跡するrank_recheck機構)を持つため、追加のロジックなしで正しく検知できる
4. 暗転 → ここでクールダウン(この試合の記録完了とみなす)
5. マッチング画面 → 次の試合の検知を再開

### データ保存

- SQLite。詳細スキーマは `README.md` のドラフトを参照(未確定、実装時に精査すること)

### `src/` のモジュール構成

```
src/
└── nss_tracker/
    ├── capture/            # ffmpegサブプロセス起動・生フレームの継続読み取り/バッファリング
    ├── detection/          # 画像解析ロジック(banner.py: 勝敗バナー判定, rank_ocr.py: ランクOCR, motion.py: ピクセル差分監視, match_end.py: 「試合終了」バナー検知)
    ├── detection_config.py # detection/配下のROI・色閾値をconfig/detection.tomlから読み込むローダー
    ├── state/              # 試合の状態遷移(バナー表示→ランクアニメ→確定→暗転→マッチング)の管理
    ├── database/           # SQLiteへの読み書き
    └── web/                # 将来のグラフ表示用(今は空、境界だけ確保)
```

- `detection/`は当面`banner.py` / `rank_ocr.py` / `motion.py` / `league_change.py` / `goal.py` / `match_end.py`のフラット構成とする。追加OCRなど将来の機能追加が必要になった段階で、その都度サブディレクトリに整理し直してよい(先回りして細分化しない)

### 検知パラメータ(ROI・色閾値)のconfig化

- `detection/`配下の各モジュール(`banner.py`, `rank_ocr.py`, `league_change.py`, `goal.py`, `motion.py`, `match_end.py`)が持つROI・HSV色閾値・ピクセル差分閾値、および`state/match_state.py`の一部の検知閾値(fpsに依存しないもの)は、ルート直下`config/detection.toml`(git追跡対象、デフォルト値入り)から読み込む。読み込みは`src/nss_tracker/detection_config.py`の`get_detection_value(section, key, default)`が担当し、各モジュールのモジュールレベル定数の初期化時に1回呼ばれる
- `config/detection.toml`はモジュールごとに`[banner]` / `[rank_ocr]` / `[league_change]` / `[goal]` / `[motion]` / `[match_state]`のテーブルを持つ。ファイル自体が無い、またはテーブル・キーが無い場合は各モジュール側のPythonデフォルト値(=元々ハードコードされていた値)にフォールバックする
  - `state/match_state.py`の`DEFAULT_BANNER_CONFIRM_FRAMES`等のフレーム数系デフォルト値は対象外(`main.py`が実際のfpsに応じて動的に再計算して上書きするため、素の値を外に出すと二重管理になる。Issue #49参照)
- fixture実測に基づく閾値決定の根拠コメントは、詳細を失わないよう各detectionモジュールのPython定数側に残す(config/detection.toml側は簡潔なコメントのみ)
- 将来キャプチャ以外のカテゴリの設定が増えた場合も、`config/`配下に種別ごとのファイルを追加していく想定(例: 将来`config/xxx.toml`)。デバイス名・解像度(Issue #30)は値がシンプルなKEY=VALUEで足りるため、従来どおり`.env`(`config.py`)で扱う
- TOML読み込みには標準ライブラリの`tomllib`を使うため、`requires-python`は`>=3.11`(2026-07時点で`>=3.10`から引き上げ済み)

### ゴール・アシストの記録とプレイヤー許可リスト

- 得点者・アシスト名は`detection/goal.py`でOCR読み取りする。数字専用の`rank_ocr.py`とは異なり任意の日本語混じり文字列を読む必要があるため、EasyOCR(信頼度0.00〜0.16程度でほぼ使い物にならなかった)ではなく**PaddleOCR**(`lang="japan"`)を使う。実測では16件中15件(94%)を正しく読み取れた
- プレイヤー許可リスト: `.env`(`.gitignore`対象、他プレイヤーの実名を含むため)の`ALLOWED_PLAYERS`にカンマ区切りで書かれたプレイヤーが得点者・アシスト者の**どちらか一方でも**含まれていれば、そのゴールを記録する(`database/db.py`の`save_goal`が担当)。得点者・アシスト者のどちらも許可リストに無い場合のみ記録すらしない。テンプレートは`.env.example`(git追跡対象)を参照すること
  - 得点者・アシスト者のうち許可リストに無い側の名前も、記録するかどうかが決まった以上はそのまま(フィルタ・匿名化せず)保存する。運用は個人のローカル環境のみのため、許可リスト外の実名がDBに残ること自体は問題にならないという判断による(2026-07時点の決め事)
- 許可リストによるフィルタリングは永続化層(`database/db.py`)の責務とし、検知層(`detection/`)・状態機械(`state/match_state.py`)はポリシーを持たず見えたものをそのまま報告する
- ゴール検知はWATCHING状態(結果バナーを待っている=プレイ中)の間のみ有効。試合単位でメモリ上にバッファし、試合結果確定時に`MatchResult.goals`として払い出す

### テスト方針

- `tests/`に`pytest`を導入し、`fixtures/screenshots/*.png`を入力に`detection/`配下の関数(banner判定・OCR・motion監視)を検証する
- ファイル名は`docs/screen_states.md`の命名規則に対応させ、パラメータ化テストで期待値(勝敗・ランク値など)を検証する
- `fixtures/screenshots/`は`.gitignore`対象でリモートに含まれないため、**fixtureディレクトリが存在しない環境ではこれらのテストを自動でskipする**(`pytest.mark.skipif`等でディレクトリの有無をチェック)
- 状態遷移(`state/`)やDB読み書き(`database/`)など実画像に依存しないロジックは合成データ/モックでテストし、CI・クローン直後でも常に実行できるようにする
- OCR(EasyOCR)は重い処理のため`@pytest.mark.slow`などで分離し、デフォルトの`pytest`実行では回さず明示的に指定したときのみ実行する

### ログ方針

- 標準の`logging`モジュールを使う(`print()`は使わない)。設定は`app.py`(エントリーポイント)で一箇所にまとめる
- 出力先はコンソール(標準出力)+ローテーティングファイル(`logs/tracker.log`、`.gitignore`の`*.log`で除外対象)の両方
- タイムスタンプは常に**JST**(日本標準時)で明記する。OSのタイムゾーン設定に依存させず、`main.py`の`_jst_converter`でUTC+9固定オフセットに変換してから出力する(ログ行に`JST`と明記、`main.py`参照)。DB側(`timeutil.now_jst`)とタイムゾーンを揃えることで、ログとDBの記録を見比べる際に時刻がズレて見えないようにしている
- モジュールごとに`logging.getLogger("nss_tracker.<モジュール名>")`(例: `nss_tracker.database`, `nss_tracker.state`)の子ロガーを使ってよい。ハンドラの設定は`main.py`の`nss_tracker`ロガーにのみ行い、子ロガーはpropagateで自動的にそこへ流れる(`main.py`側の`propagate = False`は`nss_tracker`自身がroot loggerへ二重出力しないためのものであり、子ロガーからの伝播は妨げない)
- ログレベルの使い分け:
  - `DEBUG`: フレーム単位・ピクセル差分などの高頻度な内部状態(デフォルトでは出さない)
  - `INFO`: 試合ライフサイクルの節目(バナー検知、ランク確定、記録完了など)。「検知はできたが記録しなかった」(許可リスト外のゴール、ランクバッジが読み取れなかった等の想定内の欠落)もここに含め、記録結果を見ただけでは分からない経緯を追えるようにする。ただし許可リスト外のプレイヤーの実名はログにも出さない(記録しないと決めた実名を別の場所に書き出さないため)
  - `WARNING`: 想定外の状態(フレーム取得タイムアウト、想定外の画面遷移など)
  - `ERROR`: 復旧不能な異常(ffmpegプロセスの異常終了など)
- レベル切り替えは環境変数か定数1つで簡単に済ませる(凝った設定ファイルは今は不要)

### 常駐アプリとしての起動方法

- 現段階では手動でターミナルから`uv run python main.py`を実行する運用のみを想定し、タスクスケジューラ等の自動起動・自動復旧の仕組みは作らない
- 実際に配信で運用してみて自動復旧などの必要性が具体化した段階で改めて検討する
- `main.py`でcapture(`FfmpegFrameReader`)→state(`MatchStateMachine`)→database(`db`)の配線を実装済み。OBS/Switchが無い段階でも配線を確認できるよう、`--video path/to/file.mp4`で動画ファイルをOBS Virtual Cameraの代わりに読み込める(動作確認は完了)。実機OBS Virtual Cameraでの疎通確認も完了済み(`docs/capture_verification.md`参照)。ただし`main.py`を実キャプチャ相手にend-to-endで動かし、実際に試合をプレイしてbanner検知・ランクOCR・ゴール検知が正しく発火するかの確認はまだ行っていない
  - `MatchStateMachine`のフレーム数ベースの閾値(`banner_confirm_frames`等)は30fps想定のデフォルト値のため、呼び出し側(`main.py`)で実際のfpsに応じてスケーリングする必要がある(`state/match_state.py`のdocstring参照)。`--video`指定時はファイルのfpsを自動検出、実キャプチャ時は30fps想定
  - EasyOCR・PaddleOCRは初回呼び出し時にモデル読み込みで数秒かかる(実測: 約3.8秒、2回目以降は0.2秒未満)。キャプチャループ開始前に一度呼び出して済ませておかないと、ちょうどランクバッジの安定待ちが始まる直後にこの数秒間が重なり、`FfmpegFrameReader`の「追いつかない間は古いフレームを破棄する」設計と組み合わさって肝心の区間のフレームを丸ごと読み飛ばしてしまう(`main.py`の`_warmup_ocr_engines`で対応済み)

## 開発フロー(Git運用)

詳細は [`docs/git_workflow.md`](docs/git_workflow.md) を参照。要点は以下の通り、必ず守ること。

- `main`へ直接pushしない。タスクごとに`feature/xxx`・`fix/xxx`ブランチを切って作業する
- 1タスク=1ブランチ=1PRを基本とし、`main`へのマージ前に必ずPRを作成する
  - 1タスクが大きくなるときは、親ブランチ`feature/xxx`を切り、子ブランチ`feature/xxx/yyy`をそこにマージし、最後に`main`ブランチへマージする
- コミットメッセージはConventional Commits形式(`feat:`, `fix:`, `docs:` など)でこまめに行う
- PRは見出し3以下を使用し、動作確認内容とドキュメント更新の有無を必ずチェックボックスで含める
- 新たな実装をしたり、決め事をした場合は、必要に応じて都度マークダウンファイルに追記する
- `main`・親ブランチへのforce-pushや履歴書き換え(`rebase -i`など)は行わない。必要な場合は必ずユーザーに確認する
- 上記「未確定・要相談」に該当する仕様に関わる実装は、着手前に必ずユーザーに確認を取ってから進める
