# Git / GitHub 運用ルール(ドラフト)

個人開発想定の最低限のルール。未確定事項は今後新たに出てきた場合、都度追記して決めていく。

## ブランチ運用

- `main`: 常に動く状態を保つ
- 作業用ブランチ: `feature/xxx`、`fix/xxx` のように接頭辞をつける
  - 例: `feature/rank-ocr`, `fix/banner-false-positive`
- タスクが大きくなる場合は親ブランチ/子ブランチ構成にする
  - 親ブランチ: `feature/xxx`(通常の命名と同じ)
  - 子ブランチ: 親ブランチ名をハイフンで接頭辞にする `feature/xxx-yyy`(`/`区切りは不可、後述の注意点を参照)
    - 例: 親`feature/rank-detection` → 子`feature/rank-detection-ocr`, `feature/rank-detection-schema`
  - 子ブランチは親ブランチにマージし、最後に親ブランチを`main`にマージする
  - **注意**: 子ブランチを`feature/xxx/yyy`のように親ブランチ名をそのまま`/`区切りで接頭辞にすることはできない。Gitのブランチ参照(`refs/heads/...`)はパスとして階層的に保存されるため、`feature/xxx`というブランチが存在する状態では同じ名前を親ディレクトリとする`feature/xxx/yyy`を同時に作成できない(`cannot lock ref`エラーになる、2026-07-17に実際に発生し発覚)。そのため子ブランチは親ブランチ名+ハイフンで区別する

## 開発フロー(Claude Codeでの開発)

Claude Codeでタスクに着手する際は、以下の流れに沿って進める。

1. **着手前にGitHub Issueを起票する**(ドキュメントのみの変更を除く)。実装を伴う変更は、必ず対応するIssueを作ってから着手する。詳細は「Issue / タスク管理」を参照
2. **着手前に`main`を最新化する**(`git pull`)
3. **タスクごとにブランチを切る**。`main`に直接コミットしない
   - 通常のタスク: `main`から`feature/xxx`・`fix/xxx`を切る
   - タスクが大きい場合: 親ブランチ`feature/xxx`を切り、そこからさらに子ブランチ`feature/xxx-yyy`を切って作業する
4. **こまめにコミットする**(Issue番号を含むConventional Commits形式、詳細は次項)
5. **新たな実装をしたり、決め事をした場合は、必要に応じて都度関連するマークダウンファイル(README/CLAUDE.md/docs配下)に反映する**
6. 作業が完了したら、マージ前に**PRを作成する**(通常タスクは`main`へ、子ブランチは親ブランチへ)。対応するIssueに紐付ける
   - フォーマットは次項「PRのフォーマット」に従う
7. 内容をセルフレビューしたうえでマージする(子ブランチ→親ブランチ、親ブランチ→`main`)
8. マージ後、不要になった作業ブランチは削除する。対応するIssueが完了していればクローズする(`main`へのマージでPR本文の`Closes #N`により自動クローズされる)

### PRのフォーマット

- 見出しは`###`(見出し3)以下を使用する(PRタイトルが見出し1・2相当のため)
- 本文には最低限、以下をチェックボックス付きで含める

```markdown
### 変更内容
- 何を・なぜ変更したか

### 動作確認
- [ ] 確認した動作1
- [ ] 確認した動作2

### ドキュメント
- [ ] 関連するREADME/CLAUDE.md/docsを更新した(該当する場合)
- [ ] 更新不要

Closes #<Issue番号>
```

- 本文末尾に`Closes #<Issue番号>`を含める。マージ時にGitHubが対応するIssueを自動でクローズする
- ドキュメントのみの変更などIssueに紐付かないPRでは省略してよい

### Claude Codeが必ず守ること

- `main`へ直接pushしない。作業は必ずブランチ上で行う
- **実装を伴う変更は、着手前に対応するGitHub Issueを起票する(ドキュメントのみの変更は対象外)**
- 1つの目的(1タスク)に対して1ブランチ・1PRを基本とする。関係のない変更を混ぜない。タスクが大きい場合は親ブランチ/子ブランチ構成にする
- コミットメッセージは`[Issue番号]<type>: <説明>`の形式(Conventional Commits + Issue番号プレフィックス)でこまめに行う。Issueに紐付かない変更(ドキュメントのみ等)ではIssue番号を省略してよい
- 新たな実装・決め事は都度マークダウンファイルに反映し、ドキュメントを実装より古い状態のまま放置しない
- PRは見出し3以下を使用し、動作確認とドキュメント更新有無を必ずチェックボックスで明記する。対応するIssueがあれば`Closes #N`で紐付ける
- **共有ブランチ(`main`および親ブランチ)に対して`force-push`や履歴の書き換え(`rebase -i`など)を行わない。どうしても必要な場合は、実行前に必ずユーザーに確認を取る**
- **未確定の仕様(`CLAUDE.md`の「未確定・要相談」に該当する箇所)に関わる実装は、着手前に必ずユーザーに確認する**。確認なしに仕様を決めて実装を進めない
- `fixtures/screenshots/` を誤ってコミット・pushする変更を書かない(後述の.gitignore方針を参照)
- マージ方法は**通常のマージ(merge commit)**を採用する(2026-07-19変更。以前はsquash mergeだったが、コミットグラフから作業のまとまりが分かるようにしたいというユーザーの意向による)
- 1ブランチ内のコミットは、作業の構造がはっきりしている場合(実装本体とテストが分けられる、機能実装と設定変更が分けられる等)は1つに詰め込まず、いくつか(目安2〜4個程度)に分ける。ただし色閾値のチューニング等、試行錯誤(スクラッチパッドでの検証)が主体で最終的に動くものだけをコードに落とすようなタスクでは、無理に工程を分割しない(「一旦壊れた→直した」というノイズだけの分割になり、見て意味のある単位にならないため)。過度な細分化もしない
- **PR作成前に`uv run pytest`(デフォルトの非slowテスト)を実行し、パスすることを必須とする**。OBS Virtual Camera実機・OCRなどローカル環境でしか検証できないテストが多く、CIに任せきりにできないため

## コミットメッセージ

[Conventional Commits](https://www.conventionalcommits.org/) 形式に、対応するGitHub Issueの番号をプレフィックスとして付ける。

```txt
[<Issue番号>]<type>: <説明>

例:
[25]feat: 結果バナーの色判定ロジックを追加
[26]fix: ランクOCRが暗転時に誤検知する問題を修正
[27]test: league_change.pyのテストを追加する
chore: 依存パッケージを更新
```

- ドキュメントのみの変更などIssueに紐付かないコミットは、Issue番号プレフィックスを省略してよい(上記`chore`の例)

主な `type`:

| type | 用途 |
| --- | --- |
| feat | 新機能 |
| fix | バグ修正 |
| docs | ドキュメントのみの変更 |
| refactor | 挙動を変えない内部整理 |
| test | テストの追加・修正 |
| chore | ビルド設定・依存関係など |

## Issue / タスク管理

GitHub Issuesを使用する。

- **実装を伴う変更は、着手前に対応するIssueを起票してから進める**(ドキュメントのみの変更は対象外)
- Issueには背景・やること(チェックリスト)を書く。スコープが大きい場合は着手時に分割を検討する
- コミットメッセージ・PRとIssueの紐付け方は「コミットメッセージ」「PRのフォーマット」を参照

### Issueタイトルの接頭辞

Issueが何を目的とするものか一覧でも一目で分かるよう、タイトルの先頭に種別を表す接頭辞をつける(2026-07-19導入)。

| 接頭辞 | 用途 |
| --- | --- |
| 【バグ】 | 既存動作の不具合修正 |
| 【機能追加】 | 既存に無い新しい機能の追加 |
| 【機能改修】 | 既存機能の変更・拡張 |
| 【内部改善】 | リファクタ・設定変更・fixture収集等、外部からの見え方を変えない作業 |
| 【検証】 | 実プレイ確認等、実装を伴わない確認・調査作業 |

例: `【バグ】classify_banner()がマッチング待機画面で誤検知する問題を修正する`

上記5種類で分類しづらいIssueが出てきた場合は、その都度カテゴリを追加してよい。

## .gitignore の方針

以下は含めない(リモートにpushしない)想定:

- 実機キャプチャの生録画データ(検証用の短いクリップを除く)
- 仮想環境ディレクトリ(`.venv/` など)
- `__pycache__/`, `*.pyc`
- ローカルのSQLiteデータファイル本体(スキーマやマイグレーションは含めるが、実データは含めない)
- **`fixtures/screenshots/` 配下の参照画像一式**
  - 理由: 他のプレイヤーの見た目やプレイヤー名がそのまま映り込んでいるため、プライバシー配慮でリモートには一切上げない。ローカル環境にのみ置く運用とする
  - `docs/screen_states.md` に画像のファイル名一覧(=期待される命名規則)は記載済みなので、リポジトリをクローンした環境では各自ローカルで画像を配置する
  - 空フォルダの構成だけは把握できるよう、`fixtures/screenshots/.gitkeep` など中身のないファイルのみ追跡する

- `uv.lock` は依存関係の再現性のため追跡対象(コミットする)

## リリース・タグ

バージョニング(タグ付け・バージョン番号管理)は行わない。個人開発で常に`main`の最新版を実行する運用のため、実益が薄いと判断した。DBスキーマ変更などで過去データとの互換性に影響がある場合は、タグではなく該当箇所のドキュメント(README等)にその都度明記する。将来、配布や複数バージョンの並行運用が必要になった段階で改めて検討する。
