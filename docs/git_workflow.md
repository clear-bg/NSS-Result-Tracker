## ブランチ運用

- `main`: 常に動く状態を保つ
- 作業用ブランチ: `feature/xxx`、`fix/xxx` のように接頭辞をつける
  - 例: `feature/rank-ocr`, `fix/banner-false-positive`
- タスクが大きくなる場合は親ブランチ/子ブランチ構成にする
  - 親ブランチ: `feature/xxx`(通常の命名と同じ)
  - 子ブランチ: 親ブランチ名を接頭辞にする `feature/xxx/yyy`
    - 例: 親`feature/rank-detection` → 子`feature/rank-detection/ocr`, `feature/rank-detection/schema`
  - 子ブランチは親ブランチにマージし、最後に親ブランチを`main`にマージする

## 開発フロー(Claude Codeでの開発)

Claude Codeでタスクに着手する際は、以下の流れに沿って進める。

1. **着手前に`main`を最新化する**(`git pull`)
2. **タスクごとにブランチを切る**。`main`に直接コミットしない
   - 通常のタスク: `main`から`feature/xxx`・`fix/xxx`を切る
   - タスクが大きい場合: 親ブランチ`feature/xxx`を切り、そこからさらに子ブランチ`feature/xxx/yyy`を切って作業する
3. **こまめにコミットする**(Conventional Commits形式、詳細は次項)
4. **新たな実装をしたり、決め事をした場合は、必要に応じて都度関連するマークダウンファイル(README/CLAUDE.md/docs配下)に反映する**
5. 作業が完了したら、マージ前に**PRを作成する**(通常タスクは`main`へ、子ブランチは親ブランチへ)
   - フォーマットは次項「PRのフォーマット」に従う
6. 内容をセルフレビューしたうえでマージする(子ブランチ→親ブランチ、親ブランチ→`main`)
7. マージ後、不要になった作業ブランチは削除する

### PRのフォーマット

- 見出しは`###`(見出し3)以下を使用する(PRタイトルが見出し1・2相当のため)
- 本文には最低限、以下をチェックボックス付きで含める

```markdown
### 変更内容
- 何を・なぜ変更したか

### 動作確認
- [ ] 確認した動作1
- [ ] 確認した動作2

### ドキュメント
- [ ] 関連するREADME/CLAUDE.md/docsを更新した(該当する場合)
- [ ] 更新不要
```

### Claude Codeが必ず守ること

- `main`へ直接pushしない。作業は必ずブランチ上で行う
- 1つの目的(1タスク)に対して1ブランチ・1PRを基本とする。関係のない変更を混ぜない。タスクが大きい場合は親ブランチ/子ブランチ構成にする
- コミットメッセージはConventional Commits形式でこまめに行う
- 新たな実装・決め事は都度マークダウンファイルに反映し、ドキュメントを実装より古い状態のまま放置しない
- PRは見出し3以下を使用し、動作確認とドキュメント更新有無を必ずチェックボックスで明記する
- **共有ブランチ(`main`および親ブランチ)に対して`force-push`や履歴の書き換え(`rebase -i`など)を行わない。どうしても必要な場合は、実行前に必ずユーザーに確認を取る**
- **未確定の仕様(`CLAUDE.md`の「未確定・要相談」に該当する箇所)に関わる実装は、着手前に必ずユーザーに確認する**。確認なしに仕様を決めて実装を進めない
- `fixtures/screenshots/` を誤ってコミット・pushする変更を書かない(後述の.gitignore方針を参照)

## コミットメッセージ

[Conventional Commits](https://www.conventionalcommits.org/) 形式を採用する。

```
<type>: <説明>

例:
feat: 結果バナーの色判定ロジックを追加
fix: ランクOCRが暗転時に誤検知する問題を修正
docs: CLAUDE.mdに検知方式の方針を追記
chore: 依存パッケージを更新
```

主な `type`:

| type | 用途 |
|---|---|
| feat | 新機能 |
| fix | バグ修正 |
| docs | ドキュメントのみの変更 |
| refactor | 挙動を変えない内部整理 |
| test | テストの追加・修正 |
| chore | ビルド設定・依存関係など |

## .gitignore の方針

以下は含めない(リモートにpushしない)想定:

- 実機キャプチャの生録画データ(検証用の短いクリップを除く)
- 仮想環境ディレクトリ(`.venv/` など)
- `__pycache__/`, `*.pyc`
- ローカルのSQLiteデータファイル本体(スキーマやマイグレーションは含めるが、実データは含めない)
- **`fixtures/screenshots/` 配下の参照画像一式**
  - 理由: 他のプレイヤーの見た目やプレイヤー名がそのまま映り込んでいるため、プライバシー配慮でリモートには一切上げない。ローカル環境にのみ置く運用とする
  - `docs/screen_states.md` に画像のファイル名一覧(=期待される命名規則)は記載済みなので、リポジトリをクローンした環境では各自ローカルで画像を配置する
  - 空フォルダの構成だけは把握できるよう、`fixtures/screenshots/.gitkeep` など中身のないファイルのみ追跡する

- `uv.lock` は依存関係の再現性のため追跡対象(コミットする)
